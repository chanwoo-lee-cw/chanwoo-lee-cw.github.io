# Redis

## Redis란?

> Key-Value 구조의 비관계형 In-Memory 데이터 저장소이다.

DB가 있는데도 Redis를 사용하는 이유는, 일반적인 RDB는 디스크에 데이터를 저장하기 때문에 영속성은 좋지만 조회 속도가 상대적으로 느리고, 자주 조회할수록 디스크 I/O 부하가 커지기 때문이다.

그래서 자주 조회되는 데이터를 메모리에 올려두고 빠르게 응답하기 위해 Redis가 자주 사용된다. 다만 메모리는 디스크보다 용량이 비싸고 한정적이기 때문에, 무엇을 얼마나 올릴지(그리고 언제 비울지)를 설계하는 것이 중요하다.

## 특징

### 1. 독립된 서버 프로세스로 동작한다

- Redis는 별도의 서버에 설치해두고 네트워크(IP + Port)를 통해 접근한다.
  - 예를 들어 AWS에 Redis 서버를 구축하면, 여러 대의 애플리케이션 서버가 동일한 Redis를 공유해서 사용할 수 있다.
- 즉, 애플리케이션과 분리된 독립적인 서버 프로세스로 동작한다. 이 덕분에 분산 환경에서 "공유 저장소" 역할을 할 수 있다.
### 2. In-Memory, 메모리 기반 데이터 저장소이다

- Redis는 데이터를 기본적으로 RAM에 저장한다.
  - DB보다 훨씬 빠른 응답 속도를 제공한다.
- Cache나 임시 저장소로 사용하기 적절하다.
- 단, 메모리는 한정적이므로 `maxmemory` 한도와 eviction(축출) 정책을 함께 설계해야 한다.

> [!NOTE]
> In-Memory이지만, 데이터가 무조건 휘발되지는 않는다 Redis는 RDB/AOF 설정으로 디스크에 영속화할 수 있어서 재시작 후 복구가 가능하다.

### 3. Key-Value 기반 자료구조를 사용한다

- 데이터를 Key-Value 형태로 저장하고, Key를 기반으로 Value를 찾는 구조이다.
- 내부적으로 키 공간이 해시 테이블로 관리되어 **키 조회 자체는 평균 `O(1)`** 이다.
- 단, 모든 연산이 O(1)인 것은 아니다. 자료형과 명령에 따라 복잡도가 다르다.
  - `GET`/`SET` 등 키 단위 조회: `O(1)`
  - `ZADD`(Sorted Set 삽입): `O(log N)`
  - `KEYS`, `SMEMBERS` 등 전체 순회: `O(N)` -> 즉, `SCAN`을 대신 사용한다.

### 4. 다양한 자료형을 제공한다

- Value로 단순 String만 저장하는 것이 아니라, 다양한 자료형을 제공해 유연하게 사용할 수 있다.
- 대표적인 자료형(Collection):
  - String
  - List
  - Set
  - Sorted Set(ZSet)
  - Hash
  - Stream
  - 그 외: Bitmap, Bitfield, HyperLogLog, Geospatial 등

### 5. 싱글 스레드로 명령어 처리

- 커맨드는 싱글 스레드로 처리하기때문에, 순차적으로 처리한다.
  - 즉, 단일 명령은 원자적으로 실행되서, 다른 명령이 끼어들지 않는다.
  - 싱글 스레드로 순차 처리해서 락과 컨텍스트 스위칭 비용이 없어서 처리 속도가 매우 빠르다.
- 단, 모든 명령을 싱글 스레드로 처리하는건 아니라, 무거운 작업은 따로 뺀다.
  - 네트워크 I/O(소켓 read/write)는 멀티 스레드로 처리한다.
  - `bgsave`(스냅샷), `lazyfree`(대용량 키 삭제) 같은 무거운 작업은 백그라운드 스레드에서 처리한다.

> [!IMPORTANT]
> 싱글 스레드 기반이기 때문에, 단일 명령의 원자성은 보장되지만, 여러 명령어를 거치는 로직의 경우에는 `INCR`/`DECR`(원자적 증감), `MULTI/EXEC`(트랜잭션), `WATCH`(낙관적 락), Lua 스크립트를 사용해야한다.

### 6. TTL 기능

- 키에 만료 시간(TTL)을 설정해 일정 시간 후 자동으로 삭제되게 할 수 있다.
- 캐시 만료, 세션 제한 시간, 분산 락 자동 해제 등에 핵심적으로 쓰인다.



## 영속성 (Persistence)

메모리 기반으로 데이터를 관리하기 때문에, 기본적으로 휘발성이지만, 정말로 모두 휘발성이면 한계가 있기 때문에 이에 대해  Redis는 두 가지 방식으로 디스크에 영속화할 수 있다.

| 방식 | 동작 | 장점 | 단점 |
| --- | --- | --- | --- |
| **RDB** | 특정 시점의 메모리를 스냅샷으로 저장 | 파일이 작고 복구(재시작)가 빠름 | 스냅샷 사이의 데이터는 유실 가능 |
| **AOF** | 쓰기 명령을 로그로 계속 기록 | 유실 구간이 작아 내구성이 높음 | 파일이 커지고 복구가 상대적으로 느림 |

- 둘을 함께 사용하는 하이브리드 구성도 가능하다.
- 즉, 재시작/크래시 시 데이터가 날아가는지 여부는 persistence 설정으로 방지할 수 있다. 캐시 용도면 영속성을 꺼도 되고, 영속 저장이 필요하면 RDB/AOF를 켜는 식으로 용도에 맞게 선택한다.


## 메모리 설정

Redis는 메모리를 바탕으로 데이터를 관리하기 때문에, 아주 비싸다. 그래서 관리를 해줘야 하는데 Redis는 이를 설정으로 관리할 수 있다.

- **maxmemory**: 사용할 최대 메모리 한도를 지정한다.
- **eviction(축출) 정책**: 한도에 도달했을 때 어떤 키를 밀어낼지 결정한다.
  - `noeviction`: 더 이상 쓰기를 받지 않고 에러 반환 (캐시가 아닌 저장소 용도일 때)
  - `allkeys-lru` / `allkeys-lfu`: 전체 키 중 가장 덜 쓰인 키부터 축출 (캐시 용도에 적합)
  - `volatile-ttl` / `volatile-lru`: TTL이 설정된 키 중에서 축출
- **TTL**: 키별 만료 시간으로 오래된 데이터를 만료 시켜서 삭제 시켜버린다. 

> [!TIP]
> `FLUSH`를 통해서 메모리를 비울 수도 있지만, 그보단 maxmemory + eviction 정책 + TTL으로 자동화 하는 것이 더 낫다.


## 자주 쓰이는 명령어

| 명령어                     | 설명                   | 복잡도  |
| ----------------------- | -------------------- | ---- |
| `SET key value`         | 값 저장 (옵션은 아래 참고)     | O(1) |
| `GET key`               | 값 조회                 | O(1) |
| `DEL key`               | 키 삭제 (동기)            | O(1) |
| `INCR key` / `DECR key` | 값을 1 증가/감소 (**원자적**) | O(1) |
| `INCRBY key n`          | 값을 n만큼 증가 (원자적)      | O(1) |

> `INCR`은 "조회 → +1 → 저장"을 **하나의 원자적 연산**으로 처리한다. 그래서 카운터, 조회수, Rate Limiting 등에서 락 없이 안전하게 쓸 수 있다. (앞에서 다룬 lost update가 안 생기는 이유)

**`SET`의 주요 옵션**

`NX`, `PX`는 독립된 명령어가 아니라 `SET`에 붙이는 **옵션**이다. 분산 락의 핵심 조합이라 따로 짚는다.

| 옵션 | 의미 |
| --- | --- |
| `NX` | **N**ot e**X**ists — 키가 **없을 때만** 저장 (이미 있으면 실패) |
| `XX` | 키가 **있을 때만** 저장 (NX의 반대) |
| `EX seconds` | TTL을 **초** 단위로 설정 |
| `PX milliseconds` | TTL을 **밀리초** 단위로 설정 |

**예시**

```
SET stock:productId:1 100 NX PX 3000
```
→ 키가 없을 때만(NX) 저장하고, 3초(PX) 뒤 자동 만료

**Sorted Set (ZSet) — score 기반 정렬**

> Sorted Set은 중복 없는 멤버(member)들의 집합
> 각 멤버에 score을 기준으로 항상 정렬된 상태를 유지하는 자료형


| 명령어                     | 설명              | 복잡도          |
| ----------------------- | --------------- | ------------ |
| `ZADD key score member` | score와 함께 멤버 추가 | O(log N)     |
| `ZRANGE key start stop` | 순위(인덱스) 범위로 조회  | O(log N + M) |
| `ZRANK key member`      | 특정 멤버의 순위 조회    | O(log N)     |
| `ZSCORE key member`     | 특정 멤버의 score 조회 | O(1)         |
| `ZREM key member` | 특정 멤버 제거 | O(log N) |
| `ZPOPMIN key [count]` | **score 가장 낮은** 멤버를 꺼내며 제거 | O(log N) |
| `ZPOPMAX key [count]` | **score 가장 높은** 멤버를 꺼내며 제거 | O(log N) |
| `ZREMRANGEBYRANK key start stop` | 순위 범위로 제거 | O(log N + M) |
| `ZREMRANGEBYSCORE key min max` | score 범위로 제거 | O(log N + M) |


**조회 / 순회**

| 명령어                  | 설명                | 비고                                                                     |
| -------------------- | ----------------- | ---------------------------------------------------------------------- |
| `KEYS pattern`       | 패턴에 맞는 키 전체 조회    | 전체 블로킹이라 운양에서 사용 금지<br>시간 복잡도 : O(N)                                   |
| `SCAN cursor`        | 커서 기반으로 키를 나눠서 순회 | 논블로킹이라 KEYS 명령어 대신해서 사용<br>시간 복잡도 O(N)이지만, 사실 커서 기반으로 O(1)로 조금씩 끊어서 한다 |
| `TTL key`            | 키의 남은 만료 시간(초) 조회 | 리턴 갑: <br>-1 : 만료가 설정되지 않은 키<br>-2 : 해당 키가 없다.                         |
| `EXPIRE key seconds` | 기존 키에 TTL 부여      |                                                                        |
| `UNLINK key`         | 키 삭제              | 큰 키 삭제 시 `DEL` 대신 권장<br>비동기                                            |



## 사용 시 주의점과 예시
### 주의점
- **시간 복잡도가 긴 명령러는 사용 금지**  : 대표적으로 `KEYS`, `SMEMBERS` 명령어로 보통 데이터가 수백만개 넘게 저장되는 Redis특성상 이 명령어가 들어오면 모든 요청이 블로깅 된다. 즉, 운영장애로 이어진다.
### 예시

**캐싱**
가장 자주 쓰이는 사례로 분산 환경에서 DB에서 자주 조회되는 데이터를 Redis에 캐시로 저장한다. 빠른 응답을 보장하고 TTL로 만료 시간을 설정할 수 있다.

**세션 관리**
로그인 세션을 Redis에 저장한다. 여러 서버가 동일한 Redis를 공유하므로, 사용자는 어느 서버에 붙어도 동일하게 로그인된 상태를 유지할 수 있다(세션 클러스터링). TTL로 세션 만료 시간도 관리한다.

**분산 락**
분산 환경에서 특정 자원에 대한 동시 접근을 직렬화할 때 사용한다.
보통, `SET key value NX PX <ms>` 로 원자적으로 락을 획득하고, TTL로 일정 시간 뒤 자동 해제 가능하게 설정 가능하다.

**처리율 제한 (Rate Limiting)**
"특정 사용자가 1분에 10번까지만 요청 가능"처럼 호출 횟수를 제한할 때 사용한다. 분산 환경에서 모든 서버가 동일한 카운터를 공유할 수 있고, `INCR` + TTL 조합으로 일정 시간마다 자동으로 초기화되게 구현할 수 있다.
