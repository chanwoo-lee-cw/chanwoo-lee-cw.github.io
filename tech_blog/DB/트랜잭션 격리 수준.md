# 트랜잭션 격리 수준

## 트랜잭션의 특성

아래의 4가지 특성의 앞 글자를 따서 ACID 규칙이라고 부른다.

### Atomicity (원자성)

> 한 트랜잭션의 연산은 모두 성공하거나, 모두 실패해야 한다.

하나의 트랜잭션 안에서 여러개의 연산이 있는 경우 모두 성공하거나, 모두 실패해야만 한다는 특성이 있다.

예를 들면, 클라이언트가 주문을 하였을 때 고객의 돈만 빠져나가고 주문이 안되거나, 주문이 되었는데 재고가 차감하는 것이 실패해서는 안된다.

### Consistency (일관성)

> 트랜잭션의 실행 전후로 데이터 베이스의 무결성 제약조건이 항상 만족 되어햔다.
> 트랜잭션이 완료되면 DB는 항상 일관된 상태를 유지한다.

예를 들면, 상품의 재고가 0 이상이여야 된다는 규칙이나, 특정 값은 유니크 해야한다. 같은 규칙을 일관적으로 유지해야한다는 규칙이다.

### Isolation (격리성)

> 동시에 실행되는 여러 트랜잭션이 서로 간에 영향을 주지 않아야 한다.
> 각 트랜잭션은 독립적으로 실행되는 것처럼 보여야 한다

 A라는 상품의 재고 처리를 트랜잭션1, 트랜잭션2가 동시에 처리할 때 각자 순서대로 처리한 것처럼 보여야 한다.

### Durability (지속성)

> 트랜잭션이 성공적으로 커밋되면 그 결과는 영구적으로 저장되어야 한다.

시스템 장애 등이 발생 했을 때, 완료된 트랜잭션들의 기록이 롤백되거나 복구되어서는 안된다.
즉, 한번 저장된 데이터가 사라져서는 안된다.

일반적으로 로그 등을 통해 보장한다.



## 트랜잭션 격리 수준이란?

> 여러 트랜잭션이 동시에 처리될 때, 특정 트랜잭션이 다른 트랜잭션에서 변경하거나 조회하는 데이터를 볼 수 있게 허용할지 여부를 결정하는 것



## 필요성

트랜잭션의 4가지 특성중에 Isolation(격리성)은 여러 트랜잭션이 동시에 실행될 때, 서로 간섭하지 않아야 한다는 규칙이지만, 지나치게 완벽하게 규칙을 따르려고 하면 DB의 성능이 저하된다는 문제점이 있다.

그래서, 처리 속도를 타협하기 위해서는 애플리케이션에서 요구되는 수준에 맞게 완화된 트레이드 오프를 고려해서 격리 수준을 정해야한다.



## 격리 수준 이상 현상

### Dirty Read

- 커밋되지 않는 데이터를 다른 트랜잭션이 읽는 현상. 
- 만약 작업 하던 트랜잭션이 롤백된다면 데이터의 정합성이 깨지게 된다.

| 순서 | 트랜잭션 A                   | 트랜잭션 B                                 |
| ---- | ---------------------------- | ------------------------------------------ |
| 1    | `UPDATE 잔액 = 500` (미커밋) |                                            |
| 2    |                              | `SELECT 잔액` → **500 조회**               |
| 3    | `ROLLBACK`                   |                                            |
| 4    |                              | 잔액은 다시 1000이지만 B는 이미 500을 읽음 |

### Non-repeatable Read

- 동일한 트랜잭션 내에서 두번 조회했는데, 결과가 달라지는 현상
- 작업 도중 다른 트랜잭션이 해당 행을 커밋하는 경우에 발생한다

| 순서 | 트랜잭션 A                    | 트랜잭션 B           |
| ---- | ----------------------------- | -------------------- |
| 1    | `SELECT 잔액` → **1000 조회** |                      |
| 2    |                               | `UPDATE 잔액 = 1500` |
| 3    |                               | `COMMIT`             |
| 4    | `SELECT 잔액` → **1500 조회** |                      |

### Phantom Read

- 동일한 쿼리를 반복 실행했을때, 결과 집합의 행 수가 달라지는 경우
- 다른 트랜잭션이 조건을 부합하는 새로운 행을 추가 or 삭제 하는 경우에 발생한다.

| 순서 | 트랜잭션 A                                  | 트랜잭션 B                    |
| ---- | ------------------------------------------- | ----------------------------- |
| 1    | `SELECT COUNT(*) WHERE 나이 > 20` → **5건** |                               |
| 2    |                                             | `INSERT` 나이 25인 새 행 추가 |
| 3    |                                             | `COMMIT`                      |
| 4    | `SELECT COUNT(*) WHERE 나이 > 20` → **6건** |                               |

## 4가지 격리 수준

### Level 0 - Read Uncommitted

> 커밋되지 않은 데이터도 읽을 수 있다.

- 가장 낮은 격리 수준
- 3가지 이상현상이 모두 발생할 수 있기 때문에, 정확성보다 처리량이 중요한 상황에서나 고려



### Level 1 - Read Committed

> 커밋된 데이터만 읽는다.

- Dirty Read는 방지
- 같은 트랜잭션 내 반복 조회시 다른 트랜잭션의 커밋이 반영되므로 Non-repeatable Read, Phantom Read는 여전히 발생한다.
- Oracle과 PostgreSQL의 기본 격리 수준이다.



### Level 2 - Repeatable Read

> 트랜잭션이 시작된 시점의 스냅샷을 기준으로 읽기를 수행한다.

- Dirty Read와 Non-repeatable Read는 방지
- Phantom Read는 발생할 수 있다.
- MySQL InnoDB의 기본 격리 수준이다. 하지만, InnoDB는 넥스트 키 락(Next-Key Lock)을 활용해 Phantom Read도 대부분 방지한다.



### Level 3 - Serializable

> 모든 트랜잭션이 순차적으로 실행되는 것처럼 동작한다.

- 모든 이상현상을 방지하며, 가장 높은 정합성을 보장한다.
- 단, 동시성이 줄어들고, 데드락의 발생 가능성이 높아진다.



## 서비스 당 권장 격리 수준

| 상황                                 | 권장 격리 수준   |
| ------------------------------------ | ---------------- |
| 일반적인 웹 서비스 OLTP              | Read Committed   |
| 보고서/통계 쿼리, 일관된 스냅샷 필요 | Repeatable Read  |
| 금융 거래, 재고 차감, 중복 방지 필요 | Serializable     |
| 실시간 집계, 정확도보다 속도 우선    | Read Uncommitted |
